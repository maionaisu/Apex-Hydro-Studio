## 2024-05-01 - O(N log N) cKDTree overhead vs Direct O(N) array scan for single points
**Learning:** In the `postproc_engine.py` validation method, the application was constructing a `scipy.spatial.cKDTree` (an O(N log N) operation) just to perform a single nearest-neighbor query for large unstructured meshes. This caused significant latency (e.g. 0.74s) compared to calculating the squared Euclidean distance array and finding the minimum index with `np.argmin`, which is an O(N) operation and runs drastically faster (e.g. 0.06s). Building spatial indexes like `cKDTree` is an anti-pattern when we only need to perform a single query against the data, as the overhead of building the tree is much greater than the time saved in the single O(log N) lookup.
**Action:** When searching for the nearest neighbor in large arrays for single-point queries in validation or extraction scripts, always compute the squared distance and use `np.argmin` instead of instantiating spatial index objects like `cKDTree`.
